## Fixed

- **`ourairports_get_airport` `included` field** — a new output field echoes the effective `include` selection, so a caller can tell a relation omitted by `include` (absent from `included`, its array empty for that reason) from one that genuinely has no records (present in `included` with an empty array). Text mode renders an `Included:` line and a `_Not requested_` empty-state in place of `_None recorded._` for omitted relations. ([#5](https://github.com/cyanheads/ourairports-mcp-server/issues/5))
- **`ourairports_find_navaids` unknown-code recovery** — airport mode resolves the code at the tool boundary and throws `unknown_code` carrying the declared recovery hint (mirrors `ourairports_get_airport`), replacing the service-layer `resolveAirportIdent` throw that surfaced a generic not-found with no hint. The now-dead `resolveAirportIdent` method and its orphaned `notFound` import are removed. ([#6](https://github.com/cyanheads/ourairports-mcp-server/issues/6))
- **Whitespace-tolerant code and facet inputs** — `code` (`ourairports_get_airport`, `airport://{code}`), `airport_code` (`ourairports_find_navaids`), and the `country` / `region` facets (`ourairports_search_airports`) trim surrounding whitespace at the schema boundary (`z.string().trim()`), so a padded value normalizes and a whitespace-only value fails validation instead of falling through to `unknown_code` or `mode_conflict`. ([#7](https://github.com/cyanheads/ourairports-mcp-server/issues/7))
